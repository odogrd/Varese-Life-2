## Overview

Build a **solo-admin web application** called **Varese Life**, used to scrape local events from websites, manage them, and compose a newsletter to be exported in HTML sections for use in Beehiiv. The app is entirely in **Italian** (UI labels, AI-generated content, prompts). It is a single-user system protected by email/password login.

---

## Tech Stack

- **Framework**: Node.js with Express (backend) + React (frontend, Vite)
- **Database**: PostgreSQL (persistent, stores events, newsletters, templates, sources, prompts)
- **AI**: Anthropic Claude API (claude-sonnet-4-20250514) for event text rewriting and extraction assistance
- **Scraping**: BrowserAct API (external service, REST) â€” no local scraping libraries needed
- **Editor**: TipTap (rich text WYSIWYG)
- **Styling**: Tailwind CSS â€” clean, neutral admin UI (white/light grey tones, no brand colors in UI)
- **Auth**: Simple email/password with bcrypt + express-session (single user)
- **Scheduling**: node-cron for scheduled scraping jobs
- **Error logging**: a dedicated `error_logs` table stores all application errors with full context (see Error Logging section)

### BrowserAct Integration (important)
Each source has a pre-built workflow in BrowserAct's dashboard. The app calls BrowserAct's API to run them:

```
POST https://api.browseract.com/v2/workflow/run-task
Authorization: Bearer {BROWSERACT_API_KEY}
{
  "workflow_id": "{source.browseract_workflow_id}",
  "callback_url": "{APP_BASE_URL}/api/browseract/webhook/{source.id}"
}
```

BrowserAct sends results to the webhook when the task finishes. The webhook handler:
1. Receives raw event data (JSON array)
2. For each event: saves `description_raw` to DB, then calls Claude to rewrite â†’ saves `description_clean`
3. Sets `status = 'pending'`

**Also support manual webhook/import**: A source can optionally have no `browseract_workflow_id`. In that case, show an **"Import via webhook"** endpoint URL (e.g. `POST /api/sources/{id}/import`) where BrowserAct or any external tool can POST a JSON array of events directly. This allows flexibility for sources integrated via n8n, Make, or BrowserAct's native integrations.

### Events Without a Date
If Claude cannot extract any date from the scraped or imported content, the event is **not saved** to the database. Instead it is written to the error log with the reason "Data non trovata" and the full raw content preserved, so it can be reviewed manually if needed. Events with a partially parsed date (date found but confidence is `low`) are saved normally and shown with a warning badge in the Events list.

### Multi-day Event Rule
If an event has both `date_start` and `date_end` spanning multiple days:
- The event is **assigned to the newsletter of the week containing `date_start`** (i.e. the week it begins)
- The displayed date in the newsletter uses `date_display`, automatically formatted as a range in Italian: e.g. **"4â€“6 gennaio"**, **"31 marzo â€“ 2 aprile"**
- If `date_start` and `date_end` are in the same month, format as "4â€“6 gennaio"; if they cross a month boundary, format as "31 marzo â€“ 2 aprile"
- `date_display` is auto-generated on save but fully editable by the user

### Date Parsing â€” Supported Formats

BrowserAct may return dates in any format depending on the source website. The app (and the Claude extraction prompt) must be able to parse and normalise **all** of the following into a standard `YYYY-MM-DD HH:MM:SS` timestamp stored in `date_start` / `date_end`:

**Numeric formats**
| Example | Pattern |
|---|---|
| `04/01/2027` | DD/MM/YYYY |
| `4/1/2027` | D/M/YYYY |
| `04/01/27` | DD/MM/YY |
| `04.01.2027` | DD.MM.YYYY |
| `04.01.27` | DD.MM.YY |
| `4.1.27` | D.M.YY |
| `2027-01-04` | ISO 8601 YYYY-MM-DD |
| `20270104` | YYYYMMDD compact |

**Italian text formats**
| Example | Pattern |
|---|---|
| `4 gennaio 2027` | D MMMM YYYY |
| `04 gennaio 2027` | DD MMMM YYYY |
| `4 gennaio '27` | D MMMM 'YY |
| `lunedÃ¬ 4 gennaio 2027` | dddd D MMMM YYYY |
| `lun. 4 gen. 2027` | abbreviated weekday + abbreviated month |
| `lun 4 gen 2027` | no punctuation abbreviations |
| `4 gen 2027` | D MMM YYYY abbreviated |
| `4 gen. 2027` | D MMM. YYYY abbreviated with dot |
| `il 4 gennaio 2027` | with article |
| `lunedÃ¬ 4 gennaio` | no year (assume current or next upcoming year) |
| `4 gennaio` | no year (assume current or next upcoming year) |

**With times**
| Example | Pattern |
|---|---|
| `4 gennaio 2027 alle 20:00` | Italian "alle" prefix |
| `4 gennaio 2027 ore 20:00` | Italian "ore" prefix |
| `4 gennaio 2027 ore 20.00` | dot separator for time |
| `04/01/2027 20:00` | numeric date + 24h time |
| `04/01/2027 ore 20:00` | numeric + ore |
| `4 gennaio h. 20:00` | abbreviated "h." |
| `4 gennaio, 20:00` | comma separator |

**Date ranges (multi-day)**
| Example | Pattern |
|---|---|
| `4-6 gennaio 2027` | hyphen range, same month |
| `4â€“6 gennaio 2027` | en-dash range, same month |
| `dal 4 al 6 gennaio 2027` | Italian "dal...al" |
| `dal 4 gennaio al 6 gennaio 2027` | Italian full form |
| `dal 4 gennaio al 6 febbraio 2027` | cross-month range |
| `4 gennaio â€“ 6 febbraio 2027` | en-dash cross-month |
| `4/01 â€“ 6/02/2027` | numeric cross-month |
| `fino al 6 gennaio 2027` | "until" (implies ongoing event, use as date_end) |
| `weekend del 4-5 gennaio` | contextual weekend reference |

**Relative / contextual (resolve against current date)**
| Example | Interpretation |
|---|---|
| `questo weekend` | upcoming Saturdayâ€“Sunday |
| `sabato prossimo` | next Saturday |
| `oggi` | today |
| `domani` | tomorrow |
| `ogni sabato` | recurring â€” store first upcoming instance, flag as `recurring = true` |

### Date Parsing Implementation Notes
- The `extraction_global` and per-source prompts must explicitly instruct Claude to normalise all date formats to ISO 8601 (`YYYY-MM-DD HH:MM:SS`) before returning structured JSON
- If no year is present in the scraped date, assume the **next upcoming occurrence** of that date relative to the scraping date
- If time is missing, store `00:00:00` for `date_start` and `23:59:59` for `date_end`
- If only a single date is found (no end date), leave `date_end` as null unless the event description implies a duration
- Add a `date_parse_confidence` field (`high` / `medium` / `low`) to the events table so the UI can flag events where Claude was uncertain about the date parsing â€” show a warning icon in the Events list for `low` confidence
- `date_display` is always auto-generated from the parsed `date_start`/`date_end` in clean Italian format, but remains manually overridable

## BrowserAct Integration â€” Full Flow

```
1. User clicks "Scrapa ora" on a source (or scheduled cron fires)
   â†“
2. App iterates over all active URLs in source_urls for that source
   For each URL:
   a. Determine workflow_id: use source_url.browseract_workflow_id if set,
      otherwise fall back to source.browseract_workflow_id
   b. Call POST https://api.browseract.com/v2/workflow/run-task
      with { workflow_id, callback_url: APP_BASE_URL/api/browseract/webhook/:sourceId/:sourceUrlId }
   â†“
3. BrowserAct runs each workflow in the cloud
   â†“
4. BrowserAct POSTs results to the per-URL callback endpoint
   â†“
5. Webhook handler receives array of raw events
   â†“
6. For each event:
   a. **Deduplication check**: before inserting, check if an event with the same `source_url` already exists in the DB. If it does, **update** the existing record with any new/changed fields (e.g. updated date, price, description) rather than creating a duplicate. Log what changed. If no `source_url` match, fall back to fuzzy title+date matching: if an event with the same title and overlapping date already exists (from any source), **merge** into a single record using this priority logic for the `source_url` field:
     - Prefer **official/institutional sources** (e.g. comune.varese.it, teatrodivarese.it) over newspapers or blogs
     - If no institutional source exists, prefer the source URL that returned the **most content** (longest `description_raw`)
     - All other fields are updated with the most complete available data
     - Log the merge, including which source URL was kept and which was discarded
   b. Save to DB with `description_raw`, `status = 'pending'` (or keep existing status if updating a known event)
   c. Call Claude with `rewrite_description` prompt + `description_raw`
   d. Save `description_clean` to DB
   â†“
7. Events appear in /events with status "pending"
```

**Stale event check**: once per day (run as a cron job after the main scraping schedule), the app re-scrapes any approved events whose `date_start` is within the next 3 days and whose `last_scraped_at` is older than 24 hours. If the re-scrape finds updated content (date change, cancellation notice, description change), it updates the event and flags it with a "Modificato di recente" badge in the Events list so the admin can review. If the event page returns a 404, the event is flagged as "Possibile cancellazione" without being deleted automatically.

## Two Scraping Systems

The app has two independent scraping systems. BrowserAct is the **primary** system; the Claude fallback is the **secondary** system. Both are always available and can be triggered per-source or globally.

### Primary: BrowserAct
As described in the BrowserAct Integration section above. Used when a source has a `browseract_workflow_id` configured and BrowserAct credits are available.

### Secondary: Claude Fallback Scraper
A built-in two-level crawler using Node.js `fetch` + Claude API. Used automatically when:
- A source has no `browseract_workflow_id` configured, OR
- BrowserAct returns an error / quota is exhausted

**How it works:**
```
1. Fetch the source URL HTML using Node.js fetch (with a browser-like User-Agent header)
   â†“
2. Send the HTML to Claude with the extraction_global prompt, asking it to:
   a. Identify all individual event URLs linked from this page
   b. Return them as a JSON array
   â†“
3. For each event URL found (up to a configurable max, default 20 per source):
   a. Fetch the individual event page HTML
   b. Send to Claude to extract structured event fields
   c. Apply deduplication, date parsing, and rewriting â€” same pipeline as BrowserAct
   â†“
4. Events saved to DB with source_type = 'scraped', scraper = 'claude_fallback'
```

**Limitations to document in the UI**: the Claude fallback cannot handle JavaScript-rendered pages, login walls, or infinite scroll. For these, BrowserAct is required. A warning is shown in the Sources UI if a source has no BrowserAct workflow configured: "Questo fonte usa il sistema di backup â€” alcuni siti potrebbero non essere supportati."

**Per-source scraper preference**: each source has a `preferred_scraper` field (`browseract` or `claude_fallback`). Default is `browseract`. The admin can switch a source to always use the Claude fallback (e.g. for simple static sites where BrowserAct is overkill).

## Email-Safe HTML Export Rules

When generating export HTML (for Beehiiv), follow these rules:
- Max width: 600px, centered
- All styles must be **inline** (no `<style>` blocks, no CSS classes)
- No JavaScript
- Use `<table>` for layout where needed for email client compatibility
- **No images** are included in the exported HTML â€” image fields are stored in the DB for reference only but never rendered in newsletter output
- Font stack: `Georgia, 'Times New Roman', serif` for body, `Arial, Helvetica, sans-serif` for labels/meta
- All exported HTML must include `<meta charset="UTF-8">` if a full document, or â€” since exports are HTML fragments â€” all Italian accented characters (Ã , Ã¨, Ã©, Ã¬, Ã­, Ã®, Ã², Ã³, Ã¹, Ãº, Ã¤, Ã¶, Ã¼) must be preserved as **literal UTF-8 characters**, not HTML entities, to ensure correct rendering when pasted into Beehiiv. The server must serve all API responses with `Content-Type: application/json; charset=utf-8` and the database connection must enforce UTF-8 encoding.
- Include `lang="it"` on the root element
- No `<html>`, `<head>`, or `<body>` tags in the export â€” just the inner HTML fragment so it can be pasted into Beehiiv's custom HTML block

All errors across the app are written to an `error_logs` table in the DB. This is the single source of truth for debugging scraping failures, Claude API errors, BrowserAct timeouts, and any other system issue.

### `error_logs` table

| source | text | Where the error originated: `browseract`, `claude`, `scraper_fallback`, `import_webhook`, `manual_input`, `system` |

### Error log UI
A read-only **"Log errori"** panel is accessible from the Dashboard (as a widget showing recent unresolved errors) and from a dedicated tab within Settings. It shows:
- Timestamp, source, error type, message
- Link to the related source/event if applicable
- The raw content (collapsible) so the admin can understand what was scraped
- "Segna come risolto" button to mark resolved
- Filter by: source, error type, resolved/unresolved, date range

### What gets logged
- BrowserAct task failures or timeouts (with task ID)
- BrowserAct quota/credit exhaustion (detected from API error response)
- Claude API errors or unexpected responses
- Events not saved due to missing date ("Data non trovata")
- Fallback scraper fetch failures
- Fallback scraper parse failures (Claude returned malformed JSON)
- Import webhook errors (malformed payload, unknown source)
- Any unhandled server exceptions
